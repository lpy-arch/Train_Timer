# 应用冲突原因分析

当两个类似的 HTML 应用（如 `Train_Timer` 和 `Ins_Logger`）同时打开时，当前应用发生重启，主要原因通常归结为以下几点：

1. **同源策略与 Service Worker 冲突**：
   - 两个应用如果都托管在同一个 GitHub Pages 域名下（例如 `username.github.io`），它们共享同一个“源”（Origin）。
   - 如果其中一个应用注册了作用域（Scope）为根目录 `/` 的 Service Worker，它会拦截并控制该域名下的所有请求。
   - 当你打开第二个应用时，它可能会尝试更新 Service Worker 或触发其生命周期事件，导致浏览器为了应用新的 Service Worker 而强制重新加载（Restart）当前打开的相关页面。

2. **iOS Standalone 模式（PWA）限制**：
   - 在 iOS 上，通过“添加到主屏幕”保存的应用在 Standalone 模式下运行时，如果它们共享同一个 Origin 且作用域重合，系统有时会将它们视为同一个应用的多个实例，但在切换时处理不当会导致状态丢失或重新加载。

3. **LocalStorage 键名冲突与事件触发**：
   - 同源的应用共享 `localStorage`。如果两个应用使用了相同的键名（例如都使用了 `config` 或 `cache`），一个应用的写入操作会触发另一个应用的 `storage` 事件监听器。
   - 如果代码中存在对 `storage` 事件的全局监听并执行了 `location.reload()` 或类似的重置逻辑，就会导致应用重启。

4. **内存管理**：
   - 移动端浏览器对同一域名的资源占用有严格限制。如果两个应用都占用大量内存（例如大量 Canvas 绘图或大型缓存数据），系统可能会杀掉后台应用进程，在用户切回时重新启动。

### 建议解决方案：
- **隔离作用域**：确保每个应用的 Service Worker 作用域仅限于其子目录（如 `/Train_Timer/`）。
- **唯一前缀**：为 `localStorage` 的所有键名添加应用特定的唯一前缀（如 `tt_v2_`），避免数据互相覆盖。
- **检查事件监听**：确保没有监听全局 `storage` 事件并执行重启操作。
